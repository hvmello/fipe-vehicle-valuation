Why the app is built the way it is

FIPE Vehicle Valuation — the design choices, in plain language.

The app in one sentence: you pick a type → brand → model of vehicle, and it shows how that car's price changed across the years it was made — using Brazil's official FIPE price tables. A Java/Spring backend does the maths and talks to FIPE; an Angular screen drives it. Below is what we chose and, more usefully, why — and what we chose instead of.

The big picture

How the whole thing is shaped.

Structure

One repository for both halves

In plain termsThe backend and the website live in the same project folder (a "monorepo"): backend/ and frontend/.
WhyThey're built and reviewed together and ship as one feature. Keeping them side by side means one place to clone, one history, one CI pipeline — nothing to keep in sync across two repos.
Instead ofTwo separate repositories — more overhead for a single, cohesive deliverable.
Method

Write the spec first, then the code

In plain termsThe rules (what counts as a valid answer, the edge cases) were written down in specs/ before coding, and every test points back to a numbered requirement.
WhyIt keeps the build honest: you can trace any behaviour to a rule, and the tests prove the rules still hold after changes.
Instead ofCoding first and hoping — which makes review and future changes risky.

The backend (the engine)

Java 17 + Spring Boot. These choices are about correctness, speed and safety.

Concurrency

Reactive (WebFlux), not the classic blocking style

In plain termsOne valuation has to ask FIPE for the price of each year of the car — many small calls. We fire those off at the same time instead of one-after-another, using Spring's reactive "WebFlux" toolkit.
WhyThe work is mostly waiting on the network. Doing the calls in parallel (with a sensible limit) makes a valuation noticeably faster and uses threads efficiently.
Instead ofClassic blocking Spring MVC, where each call waits for the previous one — simpler to read, but slower for this "fan-out of many calls" shape.
Money

Exact decimals for money — never "floats"

In plain termsAll prices use BigDecimal (exact decimal numbers), rounded to two places like real currency.
WhyOrdinary computer decimals (floating point) can't represent money exactly — you get tiny errors like 0.1 + 0.2 = 0.30000000000004. For a valuation app, that's unacceptable. Exact decimals avoid it entirely.
Instead ofdouble/float — fast, but wrong for cents.
Resilience

Retry the blips, surface the real failures

In plain termsFIPE's free service occasionally drops a connection. If a call fails for that kind of temporary reason, we quietly try again a couple of times.
WhyMost hiccups vanish on a second try, so the user never sees them. But we never retry a genuine "not found" — that's a real answer, not a blip.
Instead ofFailing on the first hiccup, or blindly retrying everything (which would hammer FIPE and hide real errors).
Errors

Honest, specific error responses

In plain termsA bad vehicle type returns "400 — bad request", a missing car returns "404 — not found", and a FIPE outage returns "502 — upstream unavailable", each with a clear message (a standard format called ProblemDetail).
WhyThe website (and any other caller) can react correctly, and we never leak scary internal stack traces to users.
Instead ofA generic "500 error" for everything — unhelpful and opaque.
Parsing

Read Brazilian prices the locale-correct way

In plain termsFIPE sends prices as text like "R$ 1.234.567,89" (dots for thousands, comma for decimals). We turn that into a number using Java's built-in Brazilian-number reader, and reject anything malformed.
WhyIt's precise and clear, and it won't silently misread a broken value.
Instead ofHand-rolled text chopping with regex — easy to get subtly wrong.
Security

Locked down by default

In plain termsThe API only accepts the website's address, only the safe "read" actions, and adds standard protective headers on every response.
WhySmall, sensible defaults reduce the attack surface for free. There's no login because the data is public, read-only FIPE data — nothing to protect behind a password.
Instead ofWide-open settings, or bolting on a login that guards nothing.

The caching strategy

FIPE is slow and rate-limited, and its prices only change once a month. Caching is where a lot of the "smarts" live, so here's the thinking in detail — still in plain terms.

Pattern

Caching is a wrapper around the FIPE client

In plain termsThe part that talks to FIPE doesn't know about caching at all. A separate "wrapper" sits in front of it and remembers answers (the decorator pattern).
WhyEach piece does one job. We can change caching without touching the FIPE code, and the rest of the app is none the wiser.
Instead ofSpring's @Cacheable annotation, which for this reactive style would cache the "recipe" instead of the actual answer — a subtle bug.
Two speeds

Catalog and prices are cached differently

In plain termsThe lists of brands/models (the "catalog") barely change, so we keep them for a long time. Prices change monthly, so they're kept more briefly and tagged with which month they belong to.
WhyOne-size-fits-all would either drop the stable lists too soon or hold stale prices too long. Two speeds fit each kind of data.
Instead ofA single cache with one expiry for everything.
RAM

Keep the popular cars, drop the rest (hot/cold)

In plain termsWe don't try to hold every car in memory. The cache has a size limit and automatically keeps the frequently looked-up cars while letting rarely used ones fall out — they're simply re-fetched if someone wants them later.
WhyIt uses a sensible, bounded amount of memory no matter how big FIPE is, while still being fast for the cars people actually look at.
Instead ofPre-loading the entire FIPE catalogue — huge memory use for data nobody may request.
Freshness

A new monthly table is a trigger, not just a timer

In plain termsThe app checks daily which month FIPE is publishing. The moment that changes, it throws away the old prices so the next lookups fetch the new ones.
WhyPrices change on a known event (the monthly update), so reacting to that event is fresher and cheaper than just hoping a short timer expires at the right moment.
Instead ofOnly waiting for a fixed expiry, which could serve last month's prices for hours after the update.
In memory

The cache lives inside the app (no Redis)

In plain termsThe cache is held in the running app's own memory, not in a separate shared service like Redis.
WhyFor a single instance this is the simplest thing that works well — no extra moving parts to run or maintain. Honest trade-off: the cache is lost on restart and isn't shared if you run several copies of the app. If this ever needs to scale to many instances, a shared cache (Redis) is the documented next step.

The frontend (the screen)

Angular 18. Choices here are about being easy to use and easy to maintain.

Find a car

Type to search the model, don't scroll 500 of them

In plain termsSome brands (Ford, VW) have ~500 models. Instead of a giant dropdown, you type a few letters and it filters to matches.
WhyWe measured it: the backend answers in well under a tenth of a second — the slowness was the browser drawing 500 rows. Typing shows only a handful, so it's instant, and you find a specific car faster.
Instead ofPagination (clicking through pages) or a plain 500-item dropdown — both clumsy for this many options.
State

Modern, lightweight Angular

In plain termsThe screen uses Angular's newest building blocks (standalone components and "signals" for state) and one typed service that does all the talking to the backend.
WhyLess boilerplate, clearer data flow, and the components stay simple to read and test.
Instead ofOlder, heavier patterns (NgModules, manual subscriptions, scattered HTTP calls).
Clarity

"Nothing found" looks different from "something broke"

In plain termsIf a search genuinely has no results, you get a calm informational note. If something actually failed, you get a clear error with a "try again" nudge.
WhyThey're different situations and shouldn't look the same — one is normal, one needs attention.
Instead ofOne generic red error for both.
Readability

Prices and changes shown the Brazilian way

In plain termsValues are formatted as R$ in Brazilian style, and year-on-year changes are coloured green (up) or red (down) with an arrow — not colour alone.
WhyIt matches what a Brazilian user expects, and the arrow keeps it readable for colour-blind users too.
Instead ofRaw numbers or colour-only signals.

What we deliberately did not do

Good engineering is also knowing where to stop. These are conscious "no"s, not oversights.

Scope

No shared cache, no login, no extra infrastructure

WhyA shared cache (Redis) and authentication add real complexity that this single-instance, public, read-only app doesn't need. Each is documented as a clear next step if the requirements ever grow.
Dependencies

We didn't force a risky framework upgrade

WhyA dependency scanner flags issues in Angular's build tools (not in the shipped app). The only "fix" is a breaking jump to a much newer Angular — which would be a separate, carefully-tested migration, not something to slip in here.
